## 0. 本轮目标

把当前「可跑的逻辑骨架 + 扁平控件 UI」推到两件事同时成立：

1. **玩法闭环真正可玩**：奖励成长贯通、AI Native 能进局、压力有机械效果。  
2. **观感逼近参考图**：深色赛博校园像素风，而不是 ColorRect + 默认按钮。

**执行规则：** 每完成一个阶段 → 本地验证 → **一次中文 commit**（下方给出建议 commit message）。  
不跳阶段堆功能；UI 与玩法按阶段交替推进，避免「好看但坏档」或「能玩但像原型」。

### 1.1 与现状的差距（诚实对照）

| 维度 | 现状 | 目标图 |
|---|---|---|
| 背景 | 深色网格 ColorRect | 等距像素校园 / 教室 / 招聘办公室场景 |
| 角色 | 几乎无立绘 | 像素角色 + 光环/状态特效 |
| 卡牌 | 文字按钮块 | 费用圆标 + 插图 + 类型色带 + 短描述 |
| HUD | 散落 Label/Panel | 统一斜切边框、半透明深青面板、霓虹描边 |
| 字体 | 系统默认感 | 清晰中文 UI 字（可先用开源像素/等宽中文字体） |
| 交互反馈 | 几乎无 | 选中金/橙边、hover 高亮、意图红标、压力区地效 |

### 1.2 视觉实现策略（避免一次画完拖死）

分三层，**允许前两层先交付观感，第三层可并行补**：

1. **Layout + Theme 层（必须先做）**  
   统一色板、面板 StyleBox、按钮态、顶栏/底栏布局；即使暂时用色块/简笔画，布局也要像参考图。
2. **Sprite 占位层**  
   固定格子像素图（角色、敌人、卡面 icon、建筑 marker）；可用 AI 出图再手工裁切进 `assets/sprites/`。
3. **精修层（后置）**  
   光效、粒子、场景视差、动画帧；不阻塞玩法验收。

**色板锁定（写进 Theme / CSS 变量等价物）：**

| Token | 用途 | 建议色 |
|---|---|---|
| `bg-deep` | 底 | `#0B1014` |
| `panel` | 面板半透明 | `#0F1A20` @ 85–92% |
| `border-cyan` | 常规边框 | `#00D4E0` |
| `accent-gold` | 主 CTA / 选中 | `#E8A838` |
| `danger-red` | HP / 意图 / 压力 | `#E04545` |
| `spirit-blue` | 精神 / 能量条 | `#3A8CFF` |
| `text-primary` | 正文 | `#E8F0F5` |

禁止再扩一套「另一套紫渐变 / 奶油衬线」风格。

## 3. 阶段计划（每阶段 1 次 commit）

### 阶段 A — 视觉地基：Theme + 资源目录 + 字体

**做什么**

- 重写 `src/ui/theme.tres`（色板、Button/Panel/Label/ProgressBar 样式对齐参考图边框语言）
- 建立 `assets/` 规范：`sprites/ui/`、`sprites/chars/`、`sprites/cards/`、`sprites/bg/`、`fonts/`
- 接入一款可读中文 UI 字体（开源，商用许可清晰）
- 抽出共用控件：`PixelPanel`、`StatBar`、`IconButton`（替换 emoji 齿轮占位）

**完成标准**

- 任意新屏套 Theme 后，边框/字色/按钮态一眼像「赛博校园」而非默认 Control
- README 补一句资源目录约定

**建议 commit**

```text
feat(ui): 建立像素赛博 Theme、字体与资源目录规范
```

---

### 阶段 B — 主菜单 / 选专业屏对齐参考图

**对照** `refs/03_menu_major_select.png`

**做什么**

- 菜单：左栏金边「开始游戏」+ 次级按钮；标题字形；背景用等距校园图（可先一张整图 bg）
- 选专业：三列专业卡、八维条形图、选中金边、底栏描述
- 去掉「临时自定义专业」对主路径的干扰（可收进设置或隐藏开关，避免冲淡视觉）

**完成标准**

- 静帧对比参考图：布局分区一致（标题 / 按钮列 / 三专业 / 底栏）
- 计算机/法学/医学可点选并开局

**建议 commit**

```text
feat(ui): 重做主菜单与专业选择界面以对齐参考图
```

### 阶段 C — 玩法 P0：一局成长持久化

**做什么（逻辑优先，UI 可先用现有奖励屏）**

- `GameState` 明确存：`run_hp`、`run_spirit`、`deck_card_ids`、`permanent_stats`
- `battle._create_player()` / `battle.gd` 开战读取上述状态；结束写回
- `reward.gd` 真正合并卡牌进 `deck_card_ids`；属性进 `permanent_stats`；Buff 改为下一场开战挂上
- 补给节点改写 `run_hp`，不再只改瞬时 Character

**完成标准**

- 赢一场 → 选卡 → 再战：牌组可见新增卡，属性/生命延续

**建议 commit**

```text
fix(gameplay): 贯通肉鸽牌组属性与生命跨战斗持久化
```

### 阶段 D — 探索地图 HUD + 场景对齐参考图

### 阶段 E — 玩法 P0：AI Native 进局 + 压力机械效果

**做什么**

- `map.gd`：教学楼/图书馆按权重刷 `ai_interviewer` / `paper_reviewer`（或专用 ELITE_AI 节点）
- 压力：`run_progress` 达阈值强化普通敌 / 提前解锁终局入口 / 探索时扣精神（三选一，做一种即可）

**完成标准**

- 一局内至少能自然遇到 1 次 AI Native
- 压力数字变化会改变局内行为

**建议 commit**

```text

```

## 4. 阶段依赖与建议顺序

```text
A(视觉地基)
 ├─ B(菜单/专业 UI)
 ├─ C(成长持久化)        ← 可与 B 并行，但各自单独 commit
 ├─ D(探索 UI)
 ├─ E(AI 进局+压力)
 ├─ F(战斗 UI)
 ├─ G(AI 战 UI)         ← 依赖 E
 ├─ H(动作/Boss/结算)
 ├─ I(美术替换)         ← 依赖 B/D/F/G 布局已定
 └─ J(打包)
```

**最短可演示路径（若时间紧）：** `A → C → E → F → G → J`  
（先保证「好看战斗 + 真肉鸽 + AI」，菜单/探索可后补到 B/D/I）

## 7. 明确不做（本轮仍守 MVP 边界）

- 不做联网、不做精致逐帧全套动画
- 不做关卡编辑器、不做商业化支付
- 不让 AI 自由改数值；只在 `allowed_actions` 内决策
- 不把参考图当「必须像素级复刻」——对齐的是**布局、色板、信息层级、氛围**；美术可迭代

## 8. 风险与取舍

| 风险 | 应对 |
|---|---|
| 参考图像素密度过高、外包/出图慢 | 先锁 Layout+Theme（A/B/D/F），图用简化像素块，I 阶段再换 |
| UI 大改打坏交互 | 每阶段只动目标屏；改完跑 `test_game.py` / 手测该屏 |
| 同时追玩法与美术导致两边都不完 | 严格按阶段 commit；时间紧走「最短可演示路径」 |

## 9. 给执行 Agent 的指令摘要

1. 先读本文件 + `PROJECT_SCAN.md` + `refs/` 四张图。  
2. 一次只做一个阶段；做完勾选进度板并 commit。

4. 玩法阶段以「跨战斗可感知变强 / 能遇到 AI」为验收。  
5. 不要偷偷开新大系统；缺口以 §2 为准。
