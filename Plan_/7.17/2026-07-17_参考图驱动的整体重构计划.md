# 《专业大逃杀》参考图驱动的整体重构计划

> 日期：2026-07-17
> 状态：阶段 0～11 已实施；最终回归与双平台构建完成于 2026-07-17
> 目标版本：从“可运行 MVP 原型”推进到“接近参考图表现的可玩竖切版本”
> 视觉优先级：本轮提供的 4 张 1672×941 参考图为最高标准，归档于 `Plan_/7.17/refs/`

实施结果：项目已从固定路线 MVP 推进为可移动校园竖切，完成五专业、十名敌人、核心卡牌、AI Native 精英战、奖励成长、终局与交付回归；各阶段均有独立中文提交，验收截图见 `Plan_/7.17/progress/`。

## 1. 扫描结论

当前项目已经具备可复用的玩法骨架：

- Godot 4.4、1280×720、中文桌面端；
- 5 个专业、108 张卡牌、10 个敌人、10 个事件；
- 专业选择、固定路线探索、卡牌战斗、奖励、遗物、成就、Boss 结算已经接通；
- AI Native 使用受限行动集合，Python 服务不可用时有本地规则兜底；
- JSON 数据层、`Resource` 解析层和战斗逻辑层可以继续保留。

当前项目离参考图仍有结构性差距：

1. `assets/sprites/bg/` 的四张背景仍是几何色块，占位感很强。
2. 玩家和敌人仅有 128×192 的简化静态占位角色，缺少参考图中的轮廓、姿态、光效和受击反馈。
3. 卡牌只有 6 张按类型复用的 96×96 图标，没有独立插图、专业边框和稀有度层级。
4. 探索使用 15 个节点的固定线性路线，不是参考图中的等距校园自由移动与建筑交互。
5. 主菜单和专业选择分成两个场景，尚未形成参考图中的一体式视觉构图。
6. 普通战斗和 AI Native 战虽已有功能分区，但舞台、角色、状态、卡牌、意图和侧栏的信息层级仍不够接近参考图。
7. 设置界面从战斗或探索打开后会直接返回主菜单；首回合 AI 请求时序、最大生命重复增长、测试基线过期等问题会干扰后续重构。

因此，本轮不建议推翻项目重做。推荐策略是：

> **保留数据、战斗、奖励与 AI 核心；先修稳定性，再重建视觉基础和场景表现；用一个可自由移动的校园竖切替换线性节点体验，随后补齐普通战与 AI Native 战。**

## 2. 四张参考图的强制设计锚点

### 2.1 主菜单与专业选择

对应参考图：夜景校园 + 左侧主菜单 + 右侧专业选择。

必须保留的特征：

- 夜景等距校园作为完整背景，不在背景图中烘焙任何 UI 文字；
- 左侧为纵向菜单，主按钮使用高亮金色，次按钮为深青描边；
- 右侧专业卡是主视觉焦点，选中态使用金色描边和微光；
- 专业卡展示名称、代表物、八维属性和专属技能；
- 底部展示当前专业的详细说明；
- 压力圈、存活人数、倒计时等世界观信息可以作为氛围数据，但必须明确哪些会进入实际玩法。

### 2.2 校园探索

对应参考图：白天等距校园 + 小人移动 + 建筑热点 + HUD。

必须保留的特征：

- 顶栏固定显示头像、生命、精神、天数、学分、信用点和系统入口；
- 中央是等距校园主画面，玩家角色可移动；
- 教学楼、图书馆、宿舍、食堂、操场为可交互热点；
- 角色接近热点时出现清晰的 `E 交互` 提示；
- 左上为压力圈小地图，右侧为当前目标与可选目标；
- 底栏展示移动、冲刺、交互、背包、状态、地图、系统快捷键；
- 建筑、道路、树木和前景遮挡需要有明确层次，不能只是一张完全不可交互的平面图。

### 2.3 普通战斗

对应参考图：教室舞台 + 左右角色对峙 + 底部手牌。

必须保留的特征：

- 玩家与敌人占据舞台中心，背景必须直接说明当前区域；
- 顶部显示玩家状态、战斗标题、天数/回合、背包/地图/设置；
- 敌人意图使用高对比警示条；
- 玩家增益、减益与敌人状态分列两侧；
- 手牌在底部居中，费用、名称、插图、类型和描述具有明确层级；
- 抽牌堆、弃牌堆、能量和结束回合按钮均在固定位置；
- 攻击、受击、防御、治疗、状态变化需要有动画与音效反馈。

### 2.4 AI Native 精英战

对应参考图：就业中心/面试室 + AI 敌人档案 + 行动候选。

必须保留的特征：

- 顶部使用橙色精英遭遇横幅；
- 左栏展示 AI 敌人头像、强度、适应性、特性和掉落预览；
- 右栏展示允许行动，并高亮当前选择；
- AI 敌人通过气泡表达“分析打法、调整策略”等意图；
- 舞台中央有明显压力区或环境危险反馈；
- AI Native 的外部模型决策与本地规则兜底必须呈现相同的 UI 行为，不暴露技术错误给玩家。

## 3. 视觉与素材规范

### 3.1 分辨率和像素基线

- 继续使用项目现有的 1280×720 逻辑分辨率和 16:9 构图。
- 四张参考图为 1672×941，比例接近 16:9，可按区域比例映射到 1280×720。
- 所有界面先在 1280×720 下完成，之后再验证 1600×900 和 1920×1080 拉伸。
- 保持 `nearest` 纹理过滤；角色、图标、边框尽量使用整数倍缩放。
- 不追求传统低分辨率像素块，而采用参考图中的“高细节像素插画 + 清晰 UI 像素边框”。

### 3.2 色彩和层级

沿用并收敛现有色板：

- 深色底：`#0B1014`
- 面板底：`#0F1A20`
- 青色描边：`#00D4E0`
- 金色主交互：`#E8A838`
- 红色危险/压力：`#E04545`
- 蓝色精神/能量：`#3A8CFF`
- 米白正文：`#E8F0F5`

补充规范：

- 金色只用于主 CTA、当前选择和稀有奖励；
- 红色只用于生命损失、敌人意图和压力区；
- 青色用于常规框架和信息；
- 紫色只用于 AI、异常状态或高稀有度；
- 背景必须比角色和 UI 低一个对比度层级，避免信息互相抢夺。

### 3.3 image2 / imagegen 素材使用规则

后续允许使用 image2/imagegen 生成素材，但必须遵守：

1. 生图只负责场景、角色、道具和卡面插画，不在图片里生成中文文字、按钮、数值或 HUD。
2. UI 文本、边框、进度条和状态图标必须由 Godot 组件绘制。
3. 每批素材先生成 1 张风格样板，经游戏内截图确认后再扩批。
4. 角色素材要求透明背景、统一视角、统一光源和统一像素密度。
5. 校园背景优先拆为背景层、可行走层、前景遮挡层；无法直接拆层的生图不能直接作为最终探索图。
6. 每个素材记录用途、尺寸、提示词版本和人工修正情况，避免后续无法复现。
7. 生成后必须检查手部、文字残影、重复结构、透视错误和边缘脏像素。

### 3.4 首批素材清单

第一批只服务“计算机专业竖切”，避免一开始扩散到全部 108 张卡：

- 夜景校园主菜单背景 1 张；
- 白天等距校园探索图 1 套，至少拆成背景/前景两层；
- 教室普通战背景 1 张；
- 就业中心 AI 面试战背景 1 张；
- 计算机专业玩家：探索四方向移动、战斗待机/攻击/受击；
- 绩点焦虑者：待机/攻击/受击；
- AI 面试官：待机/分析/攻击/受击；
- 计算机初始卡组独立卡面 8～12 张；
- 通用 UI 图标、建筑热点和卡牌类型图标一套；
- 九宫格面板、按钮、标签、意图条、卡框和状态框。

竖切通过后，再按“法学 → 医学 → 金融 → 艺术”的顺序扩展角色和卡面。

## 4. 技术路线

### 4.1 保留的模块

以下模块原则上只做修复和接口整理，不推翻：

- `src/autoload/config.gd`
- `src/autoload/game_state.gd`
- `src/logic/battle.gd`
- `src/logic/card_effect_processor.gd`
- `src/logic/reward.gd`
- `src/logic/event_handler.gd`
- `src/ai/` 与 `src/server/`
- `data/` 下的专业、卡牌、敌人和事件数据

### 4.2 探索场景的最小可行实现

不建议第一步就制作完整等距 TileMap。参考图更适合下面的低风险路径：

1. 使用高质量静态校园底图作为视觉主体；
2. 在底图上配置可行走多边形和建筑碰撞；
3. 玩家使用 `CharacterBody2D`、四方向/八方向移动和 `Camera2D`；
4. 建筑入口使用统一 `Interactable` 区域；
5. 树冠、门廊等关键区域拆为前景遮挡层；
6. 通过 `YSort` 或显式层级处理玩家与场景前后关系；
7. 进入建筑或战斗时继续复用现有场景切换和数据逻辑。

这条路线能先达到参考图中的“可移动、可交互校园”观感，同时避免立刻承担完整 TileMap、寻路和大量地图拼接成本。

### 4.3 UI 组织

- 世界层和 HUD 使用不同层级，HUD 放入 `CanvasLayer`。
- 提取头像状态条、顶部资源栏、快捷键条、意图条、卡牌、状态列表等复用控件。
- 普通战和 AI Native 战共享战斗核心，但使用不同侧栏组合。
- 设置界面改为覆盖式暂停层，或记录来源场景后返回，不再直接丢失当前战斗。
- 视觉逻辑与战斗数值逻辑分开，避免继续扩大当前战斗界面脚本。

## 7. 优先级和停止条件

### 第一优先级

- 阶段 0～5：稳定基础、视觉规范、正式素材、主菜单、可移动探索和探索 HUD。

完成阶段 5 后，项目应第一次真正呈现参考图的核心形态。如果此时探索画面和交互仍不成立，应暂停后续内容扩充，先调整视觉方向。

### 第二优先级

- 阶段 6～9：普通战、卡牌表现、AI Native 战和局内成长闭环。

完成阶段 9 后，应具备一条可公开演示的完整竖切。

### 第三优先级

- 阶段 10～11：五专业内容扩批、优化和交付。

在竖切未经确认前，不批量生成 108 张独立卡面，也不批量制作全部角色动画，避免素材方向错误造成返工。

## 8. 主要风险

1. **探索范围失控**
   先做单张校园地图 + 热点交互，不立即扩展为大型无缝世界。

2. **生图素材不可拆层**
   生图前先确定背景/前景分层要求；无法拆层的图片只用于菜单或战斗背景。

3. **像素风格漂移**
   每批只生成一张样板，锁定视角、线条、光源、色板和角色比例后再扩批。

4. **UI 被背景吞没**
   背景降低饱和度和对比度，UI 使用统一暗底、青色描边和有限高亮色。

5. **功能重构引发旧逻辑回归**
   先完成阶段 0；后续视图只消费现有逻辑信号，不在 UI 中复制战斗规则。

6. **卡面工作量过大**
   先完成计算机竖切和核心卡牌，再按专业批量扩展；未确认视觉前不做全卡池。

7. **参考图中的氛围数据被误当真实系统**
   “存活人数、倒计时、压力区”等字段在开发前必须确定是纯氛围还是实际规则，不能只做假数字长期保留。

## 9. 最终完成标准

完成本计划后，玩家应能：

1. 在参考图风格的夜景校园主界面选择五个专业之一；
2. 操作像素角色在等距校园中移动并与五类建筑互动；
3. 在教室场景中完成信息清晰、反馈完整的普通卡牌战；
4. 在就业中心体验具有档案、候选行动和策略气泡的 AI Native 精英战；
5. 通过奖励、牌组、属性、遗物和压力圈完成一局成长；
6. 击败终局 Boss，看到完整结算和成就；
7. 全流程不再出现几何背景、火柴人角色或全卡池共用的类型占位图。
